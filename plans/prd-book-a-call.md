## Problem Statement

Сейчас в репозитории есть только скелет приложения «Book a call»: монорепозиторий с пакетами `api`, `backend`, `frontend`, Fastify, отдающий статический фронтенд, пустая Drizzle-схема, пустой TypeSpec-неймспейс и страницы-заглушки в React. Бизнес-сценариев из README не существует — владелец календаря не может управлять типами встреч, гость не видит доступных слотов и не может забронировать звонок. Нужно реализовать продукт согласно описанию в README так, чтобы он работал end-to-end: от выбора типа встречи и слота гостем до просмотра предстоящих встреч владельцем.

## Solution

Реализовать минимальный, но законченный сервис бронирования звонков по мотивам Cal.com:

- Владелец календаря через админ-страницу управляет типами встреч (название, описание, длительность в минутах).
- Гость на публичной странице видит список типов встреч, выбирает один, открывает календарь с свободными 30-минутными слотами и бронирует слот, указав имя и email.
- Бэкенд гарантирует, что на одно и то же время невозможно создать две брони, даже для разных типов событий.
- Владелец на админ-странице видит единый список всех предстоящих встреч по всем типам.

Авторизация отсутствует: владелец один и заранее задан, админ-страница доступна без логина. Внешние интеграции, уведомления и отмена брони не поддерживаются.

## User Stories

1. Как владелец календаря, я хочу создать новый тип встречи с названием, описанием и длительностью в минутах, чтобы гости могли выбрать его и забронировать.
2. Как владелец календаря, я хочу видеть список всех созданных типов встреч на админ-странице, чтобы понимать, что сейчас опубликовано.
3. Как владелец календаря, я хочу отредактировать существующий тип встречи (название, описание, длительность), чтобы поддерживать его актуальность.
4. Как владелец календаря, я хочу удалить тип встречи, чтобы он больше не отображался гостям.
5. Как владелец календаря, я хочу видеть единый список всех предстоящих встреч по всем типам с временем, длительностью, типом события и контактами гостя, чтобы планировать свой день.
6. Как гость, я хочу открыть публичную страницу со списком типов встреч и видеть для каждого название, описание и длительность, чтобы выбрать подходящий формат.
7. Как гость, я хочу выбрать тип встречи и открыть календарь с доступными датами, чтобы найти удобный день.
8. Как гость, я хочу видеть для выбранной даты только свободные 30-минутные слоты, чтобы не пытаться забронировать занятое время.
9. Как гость, я хочу выбрать слот и ввести своё имя и email, чтобы оформить бронирование.
10. Как гость, я хочу получить подтверждение успешного бронирования на экране, чтобы быть уверенным, что встреча создана.
11. Как гость, я хочу получить понятную ошибку, если выбранный слот уже забронирован между моментом открытия календаря и подтверждением, чтобы выбрать другой.
12. Как система, я хочу отклонять любую попытку создать бронирование на время, которое пересекается с уже существующим бронированием (любого типа), чтобы гарантировать правило занятости.

## Implementation Decisions

**API контракт (TypeSpec → openapi-typescript):**

- `EventType { id, name, description, durationMinutes }`
- `Booking { id, eventTypeId, startsAt, endsAt, guestName, guestEmail }`
- `GET /event-types` — публичный список типов (для гостя и для админа).
- `POST /event-types` — создание типа (админ).
- `PATCH /event-types/{id}` — редактирование типа.
- `DELETE /event-types/{id}` — удаление типа.
- `GET /event-types/{id}/slots?date=YYYY-MM-DD` — список свободных 30-минутных слотов в указанную дату для данного типа.
- `POST /bookings` — создание бронирования (eventTypeId, startsAt, guestName, guestEmail).
- `GET /bookings` — список предстоящих бронирований (для админа), сортировка по времени.

**Модули backend (Fastify + Drizzle + SQLite):**

- Drizzle-схема с двумя таблицами: `event_types` и `bookings`. У `bookings` — внешний ключ на `event_types`, поля времени начала/окончания, имя и email гостя. На колонке времени начала бронирования — уникальный индекс, дающий гарантию правила занятости на уровне БД.
- Drizzle Kit миграции, генерация и применение через существующие npm-скрипты.
- Слой репозитория (доступ к БД) и слой роутов Fastify, регистрирующих эндпоинты под префиксом `/api`.
- Сервис вычисления свободных слотов: для запрошенной даты строит сетку 30-минутных слотов в рабочем окне, исключает те, что пересекаются с длительностью существующих бронирований (любых типов событий). Длина слота гостя определяется `durationMinutes` выбранного типа, но сетка генерации идёт с шагом 30 минут.
- Транзакционное создание бронирования с повторной проверкой занятости и обработкой ошибки уникального индекса → 409 Conflict.
- Базовая валидация входных данных (формат email, положительная длительность, кратность 30 минутам) на уровне роутов.

**Модули frontend (React + Mantine + React Router):**

- Типизированный API-клиент, сгенерированный из openapi через `openapi-typescript`.
- `LandingPage` — список типов встреч с переходом на страницу бронирования конкретного типа.
- `BookCallPage` — выбор даты в Mantine-календаре, отображение свободных слотов, форма ввода имени и email, отправка бронирования и экран подтверждения/ошибки.
- `AdminPage` — две секции: управление типами событий (CRUD-формы и список) и список предстоящих встреч.
- Общий слой загрузки и обработки ошибок API.

**Архитектурные решения:**

- Источник правды для расписания — БД; фронтенд не хранит состояние занятости.
- Все вычисления времени делаются на бэке в UTC, фронтенд отображает в локальной таймзоне браузера.
- Fastify в production отдаёт собранный фронтенд статикой; в dev — два процесса (Vite + Fastify).
- Никакой авторизации: `/admin` доступен любому, кто знает URL (по требованию README).

## Out of Scope

- Регистрация, логин, роли, пароли, защита `/admin`.
- Поддержка нескольких владельцев календаря.
- Интеграции с внешними календарями (Google Calendar, Outlook, iCal).
- Email- или иные уведомления гостю и владельцу.
- Отмена, перенос или редактирование уже созданных бронирований.
- Настройка рабочих часов, выходных, праздников и таймзон владельца через UI.
- Произвольный размер слота отличный от 30 минут (длительность типа события может быть любой кратной 30, но сетка слотов фиксирована — 30 минут).
- Любая модификация `.github/workflows/hexlet-check.yml` и иной автогенерируемой инфраструктуры Hexlet.
