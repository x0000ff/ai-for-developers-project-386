# Book a call — Implementation Steps

Разбиение PRD `prd-book-a-call.md` на последовательные шаги. Каждый шаг самодостаточен: после его завершения репозиторий остаётся в рабочем и проверяемом состоянии (`pnpm typecheck`, `pnpm lint`, `pnpm test` и `pnpm build` зелёные). Шаги с БД начинаются с TypeSpec-контракта и включают тесты в том же шаге.

Общее правило для всех шагов с API/БД:
1. Описать контракт в `packages/api/src/main.tsp`.
2. Запустить `pnpm --filter @app/api build`, чтобы сгенерировать TS-типы.
3. Добавить/обновить Drizzle-схему и миграцию.
4. Реализовать сервис и Fastify-роут.
5. Написать тесты (unit на сервис + integration на роут через in-memory SQLite).
6. Подключить фронтенд (если шаг его затрагивает).

**Цель:** подготовить инфраструктуру для последующих шагов так, чтобы все дальнейшие шаги могли добавлять тесты без отдельной настройки.

**Работа:**

- Добавить минимальный smoke-тест «GET /health → 200» и соответствующий роут в backend, чтобы тесты проверяли живой конвейер.
- Обновить `AGENTS.md` командой `pnpm test` (если документация команд живёт там).

**TypeSpec/БД:** в этом шаге ещё нет доменных сущностей — только `/health`, поэтому без БД.

**Состояние после шага:** `pnpm install`, `pnpm build`, `pnpm test` зелёные; есть `/health`, тестовый конвейер готов.

## Step 1 — Event Types: contract, storage, CRUD API

**Цель:** владелец может через API полностью управлять типами встреч.

**TypeSpec (первым делом):**
- Модель `EventType { id: string; name: string; description: string; durationMinutes: int32 }`.
- Модели запросов `CreateEventTypeRequest`, `UpdateEventTypeRequest` (частичная).
- Операции: `GET /event-types`, `GET /event-types/{id}`, `POST /event-types`, `PATCH /event-types/{id}`, `DELETE /event-types/{id}`.
- Сгенерировать TS-типы (`pnpm --filter @app/api build`).

**БД (Drizzle):**
- Таблица `event_types` (id TEXT PK, name TEXT NOT NULL, description TEXT NOT NULL DEFAULT '', duration_minutes INTEGER NOT NULL CHECK > 0, created_at, updated_at).
- Миграция, команда `pnpm --filter @app/backend db:migrate` выполняется в тестовом bootstrap.

**Backend:**
- Репозиторий `eventTypesRepo` (list/getById/create/update/delete).
- Fastify-плагин `eventTypesRoutes` под префиксом `/api`, валидация через TypeSpec-сгенерированные схемы или ручные JSON-schema.
- Обработка 404 для несуществующего id, 400 для невалидной длительности.

**Тесты (в этом же шаге):**
- Unit-тесты репозитория на временной SQLite.
- Integration-тесты роутов: create → list → get → update → delete, плюс негативные кейсы (404, 400).

**Frontend:** не трогаем.

**Состояние после шага:** CRUD типов встреч работает через HTTP, покрыт тестами, фронтенд как был.

## Step 4 — Slots endpoint: contract, service, tests

**Цель:** есть HTTP-эндпоинт, возвращающий свободные будущие слоты для пары (тип встречи, дата).

**TypeSpec (первым делом):**
- Модель `Slot { startsAt: utcDateTime; endsAt: utcDateTime }`.
- Операция `GET /event-types/{id}/slots?date=YYYY-MM-DD` → `Slot[]`.
- Регенерация TS-типов.

**БД:** новой таблицы нет, но сервису понадобится читать будущие бронирования — поэтому в этом шаге добавляется минимальная таблица `bookings` (id, event_type_id nullable FK, event_type_name, starts_at, ends_at, guest_name, guest_email, created_at) и миграция. CRUD/создание — в следующем шаге, здесь только схема и селектор «пересекающиеся интервалы в дате X».

**Backend:**
- `slotsService.listAvailable(eventTypeId, date, now)`: строит сетку `[dayStart, dayEnd)` в рабочем окне (константы в конфиге), длиной и шагом `durationMinutes`, вычитает пересечения с `bookings`, фильтрует `startsAt > now`.
- Роут `GET /api/event-types/:id/slots` с валидацией `date`.
- 404, если `eventTypeId` не существует.

**Тесты:**
- Unit: пустая БД → полная сетка; бронь посередине дня → соответствующие слоты отсутствуют; бронь, пересекающая несколько слотов → исключены все; `now` в середине дня → прошедшие слоты не возвращаются; разные длительности типа → разный шаг.
- Integration: HTTP 200 + JSON-контракт, 404, 400 на кривой `date`.

**Frontend:** не трогаем.

**Состояние после шага:** `/api/event-types/:id/slots?date=...` отдаёт корректные слоты; таблица `bookings` создана и пуста; всё под тестами.

## Step 5 — Bookings creation: contract, API, tests

**Цель:** есть HTTP-эндпоинт для создания бронирования с соблюдением правил занятости и «только в будущем».

**TypeSpec (первым делом):**
- Модель `Booking { id; eventTypeId: string | null; eventTypeName: string; startsAt; endsAt; guestName; guestEmail }`.
- `CreateBookingRequest { eventTypeId: string; startsAt: utcDateTime; guestName: string; guestEmail: string }`.
- Операции: `POST /bookings`, `GET /bookings` (список предстоящих для владельца, сортировка по `startsAt`).
- Ответы: 201 с `Booking`, 409 при коллизии, 400 при прошедшем времени / невалидных данных, 404 при неизвестном `eventTypeId`.
- Регенерация TS-типов.

**БД:**
- Дополнить миграцию из Step 4: уникальный индекс по `starts_at`, FK `event_type_id → event_types(id) ON DELETE SET NULL`.
- Если Step 4 уже зафиксировал миграцию, добавить новую миграцию поверх.

**Backend:**
- `bookingsService.create`: в транзакции подтянуть `EventType`, вычислить `endsAt = startsAt + durationMinutes`, проверить `startsAt > now`, проверить отсутствие пересечений (`NOT (endsAt <= existing.startsAt OR startsAt >= existing.endsAt)`), записать бронь со снимком `eventTypeName`. Ловить нарушение уникального индекса → 409.
- `bookingsService.listUpcoming`: `startsAt > now`, сортировка ASC.
- Роуты `POST /api/bookings`, `GET /api/bookings`.

**Тесты:**
- Unit на сервис: успешное создание; конфликт при пересечении (разные типы, разные длительности); прошедшее время → 400; неизвестный `eventTypeId` → 404; иммутабельность (после создания брони изменение `durationMinutes` типа не меняет `endsAt` брони); удаление типа → `eventTypeId = null`, `eventTypeName` остаётся, слот брони по-прежнему занят для новых запросов.
- Integration: полный цикл POST → GET.

**Frontend:** не трогаем.

**Состояние после шага:** API полностью поддерживает цикл бронирования, тесты покрывают правила занятости, иммутабельность и ON DELETE SET NULL.

## Step 6 — BookCallPage: выбор типа, календарь, слоты, форма

**Цель:** гость проходит весь сценарий бронирования в UI.

**Работа:**
- `BookCallPage` переписывается:
  - Селектор типа встречи (предзаполнен из `?eventTypeId=`, если есть).
  - Mantine `DatePicker` для выбора даты (future only).
  - Загрузка слотов из `/event-types/{id}/slots` при смене типа/даты, индикатор загрузки, пустое состояние.
  - Выбор слота, форма `guestName` + `guestEmail`, валидация.
  - Сабмит `POST /bookings`: экран подтверждения при 201; для 409 — сообщение «слот только что заняли» и перезапрос слотов; для 400/404 — понятные ошибки.
- Переключение типа пересчитывает сетку (новая длительность).

**Тесты:** компонентные тесты ключевых состояний (список слотов, сабмит, обработка 409) с моками API-клиента.

**TypeSpec/БД:** без изменений.

**Состояние после шага:** гость может забронировать звонок end-to-end из браузера.
